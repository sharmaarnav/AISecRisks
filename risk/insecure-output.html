Insecure output handling

OWASPNISTCISAGoogle SAIF
3 controls mapped

Description

Model outputs are passed unsanitised to downstream components — web renderers, SQL engines, shell interpreters, or code executors — causing XSS, SQL injection, command injection, or SSRF via AI-generated content.

Attack scenario

An AI coding assistant generates SQL based on natural language. The application passes the output directly to a database query without parameterisation. An attacker prompts for a query that drops tables.

Real-world examples

Multiple CVEs in AI coding assistants that pass LLM-generated code directly to execution. XSS via LLM-generated Markdown in chat interfaces.

3 mapped controls

OWASP
Treat outputs as untrusted
LLM02 – Insecure Output Handling
Never pass raw LLM output to system shells, SQL engines, or code interpreters. Apply the same sanitisation you would apply to external user input.
NIST
Output classification layer
MEASURE 2.10 – Output validation
Implement output classifiers and content safety filters. Log all outputs for post-hoc review. Apply allowlisting where possible rather than blocklisting.
CISA
Context-specific encoding
Secure-by-Design Principle 2
Apply context-aware output encoding: HTML encode for web rendering, parameterise for SQL, escape for shell. Use templating engines with auto-escaping.
